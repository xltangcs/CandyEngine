## 打包系统 & 虚拟文件系统（VFS）实施状态盘点
- **PakTool**（`PakTool/Source/main.cpp`）：独立 ConsoleApp，已实现 `pack`/`list` 子命令，二进制格式 v1（MAGIC 'PAK\0' + VERSION + 条目表 + 拼接数据），无压缩/加密/过滤/资源元数据。已接入 premake 可构建。
- **PakFile**（`Candy/Core/PakFile.{h,cpp}`）：读取端，与 PakTool 格式一致，`Open/HasFile/ReadFile`。
- **FileSystem（VFS）**（`Candy/Core/FileSystem.{h,cpp}`）：单例，`Mount` 同时支持目录和 `.pak`（按扩展名识别），`Resolve` 最长前缀匹配、后挂覆盖先挂。
  - 挂载点仅在 `Application.cpp`：`/engine`→`../Candy/Content`、`/project`→`<project>/Content`（均目录）。
  - 唯一消费者是 `Renderer2D.cpp`（3 个 shader 走 `/engine/Shaders/*.glsl`）。
- **缺口/风险**：
  1. `.pak` 读取能力已实现，但运行时无任何 `Mount(".pak")` 调用 → PakTool 产出包引擎端尚消费不了（打包与 VFS 未端到端打通）。
  2. VFS 仅 shader 在用，纹理/字体/音频/场景/脚本仍裸 `std::filesystem` 路径。
  3. `Write` 只落磁盘，无缓存/热重载。
  4. ⚠️ 跨平台隐患：`Renderer2D_Quad.glsl`（大写D）请求 vs 实际 `Renderer2d_Quad.glsl`（小写d），已通过重命名文件修复。

## Game Packaging / Build（"打包"=构建独立可运行游戏 exe）
**前置误解纠正**：最初将"打包"理解为 content → .pak → VFS，实际用户要求的是"构建一个完整能玩的游戏 .exe"（类 UE Package Project）。

### ProjectSettings
- 新增 `GameProjectName` 字段（YAML 序列化/反序列化），指定要构建的 premake 项目名（如 "CandyGame"）。
- `ProjectSettingsPanel` 新增 "Game Project" 输入框。

### 编辑器菜单
- **顶级 `Build` 菜单**（与 File / Setting 同级），内含 `Build and Package...` 菜单项（非 Setting 子项）。

### EditorLayer::BuildGame() — 5 步打包流程
1. 读取 `ProjectSettings::GameProjectName`，验证非空且有已打开工程。
2. 调用 `msbuild CandyEngine.sln /t:<GameProject> /p:Configuration=Dist` 构建。
3. 从 `bin/Dist-windows-x86_64/<GameProject>/` 复制 .exe + .dll 到 `<project>/Build/`。
4. 引擎内容 → `Build/Content/`（目录 + `.pak`）；工程内容 → `Build/<ProjectName>/Content/`（目录 + `.pak`）；复制 `.candyproj`。
5. 生成 `Build/Launch.bat` → `start "" "CandyGame.exe" "<ProjectName>/<ProjectName>.candyproj"`。

### Application 内容挂载适配 standalone
- `/engine`：优先 `../Candy/Content`（编辑器），fallback `Content/`（独立发行，与 exe 同级）。`.pak` 优先于目录。
- `/project`（LoadProject）：`.pak` 优先于目录。

### 附带保留
- `PakFile::Pack` 静态方法（VFS infra，被 BuildGame 调用）。
- `Renderer2d_Quad.glsl` → `Renderer2D_Quad.glsl` 大小写修复（跨平台正确性）。

## 第三轮修复：msbuild 不在 PATH 导致 Build 失败
- 用户实测报错：`'msbuild' 不是内部或外部命令`（编辑器非从 VS Developer Prompt 启动，msbuild 不在 PATH）。
- 新增文件静态函数 `FindMSBuild()`（EditorLayer.cpp）：
  - 优先用 `vswhere.exe`（`%ProgramFiles(x86)%\Microsoft Visual Studio\Installer\vswhere.exe`）`-latest -requires Microsoft.Component.MSBuild -find "MSBuild\**\MSBuild.exe"` 取完整路径（_popen 捕获输出）。
  - 兜底扫描 VS2019/2022 的 Community/Professional/Enterprise/BuildTools 下 `MSBuild\Current\Bin\MSBuild.exe`（含 `C:\Program Files\` 与 `(x86)` 两种根）。
- `BuildGame()` Step1 改为用 `FindMSBuild()` 完整路径（带引号）调用，找不到则报清晰错误。
- 确认 premake `architecture "x64"` 但输出 token 为 `x86_64`，单架构 → MSBuild 默认 x64 平台，输出目录 `bin/Dist-windows-x86_64/<Game>/` 与 binDir 一致，无需显式 `/p:Platform=`。
- 注：直接调用 VS 安装内的 MSBuild.exe 不需 Developer Prompt 的 PATH 即可编译 C++（MSBuild 自行解析 VC 工具链）。
- 仍**未编译验证**（沙箱限制）。
