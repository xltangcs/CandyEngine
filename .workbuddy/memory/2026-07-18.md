## 用户自行实施代码后的 Review（发现 3 个严重 bug）

### Bug 1（严重）：ClearAttachment(1,-1) 被 RenderSceneTo 内部 Clear 覆盖 → 鼠标拾取失效
- 位置：EditorLayer.cpp OnUpdate L121-124, L136/143/149
- 顺序：Bind → ClearAttachment(1,-1) → RenderSceneTo(内部 Bind + SetClearColor + Clear + 渲染)
- RenderSceneTo 内部的 glClear(GL_COLOR_BUFFER_BIT) 会清所有 draw buffers（含 RED_INTEGER attachment 1），用 float(0.1,0.1,0.1,1) 清 R32I 整数缓冲 → 截断成 0，覆盖掉 ClearAttachment 设的 -1
- 后果：ReadPixel 永远读到 0 → m_HoveredEntity 永远是 entity 0 或无效 → hover/点击选中功能坏
- 修复：ClearAttachment 必须在 Clear 之后、渲染之前。最干净是把 Clear 从 RenderSceneTo 拿出来由调用方控制顺序，或 RenderSceneTo 拆 Clear + RenderScene 两步

### Bug 2（严重）：传给 RenderUITo 的 my 是 Y 翻转后的坐标 → UI 按钮命中检测 Y 反
- 位置：EditorLayer.cpp OnUpdate L154-158, L173
- L154 `ImGui::GetMousePos()` 返回左上原点；L155-156 减视口原点仍左上原点；L158 `my = viewportSize.y - my` 翻转成左下原点（给 ReadPixel 用，正确）
- 但 L173 把翻转后的 my 传给 RenderUITo → BeginGameUI 设 io.MousePos。ImGui IO.MousePos 需要左上原点，传成左下原点 → 命中检测 Y 上下颠倒
- 修复：RenderUITo 用未翻转的 my（单独算 `float uiMy = ImGui::GetMousePos().y - m_ViewportBounds[0].y;`）

### Bug 3（严重）：Simulate 态物理被跳过
- 位置：EditorLayer.cpp OnUpdate L139-144
- Scene::OnUpdateSimulation（Scene.cpp:263-289）除了 RenderScene 还做 Box2D Step + transform 同步
- 用户用 `GameFrameRenderer::RenderSceneTo(..., &m_EditorCamera)` 替换 `OnUpdateSimulation(ts, camera)`，RenderSceneTo 内部只调 `scene.RenderScene(camera)` → 物理逻辑丢失
- 注：OnUpdateEditor（Scene.cpp:291-295）只调 RenderScene，Edit 态替换无问题
- 修复：Scene 拆 OnUpdateSimulation 成 Logic（物理）+ RenderScene；Simulate 态调 Logic + RenderSceneTo

### 潜在问题 4：ImGui_ImplOpenGL3 对两个 context 都 Init/Shutdown

- 取决于 imgui 版本：若 backend data 是 per-context（新版 imgui），正确；若全局（旧版），二次 Init/Shutdown 可能有问题

## Bug 6（UI 消失真根因）：editorContext 加载 Saved/imgui.ini 覆盖游戏 UI
- 现象：打包游戏刚开始能看到 UI，放大/拖动窗口后 UI 消失（用户确认 100% DPI、打包游戏、场景等比例拉伸）
- 用户确认 `Build/Saved/imgui.ini` 存在 → 根因确认
- 根因链：
  - ImGuiLayer::OnAttach 给 editorContext 设 `io.IniFilename = "Saved/imgui.ini"`
  - 游戏工作目录 Build/ 下有 Saved/imgui.ini（运行残留）

  - Application::Run 每帧调 ImGuiLayer::Begin/End，editorContext 产生非空 draw data（Pass 3）
  - Pass 3 画到 framebuffer 0，覆盖 Pass 2 的游戏 UI
  - 放大窗口触发 ImGui 重新布局，窗口铺满屏幕，UI 被完全盖住
- 附带发现：GameLayer::OnEvent 没调 OnViewportResize → 场景相机 aspect 卡初始值 → 放大窗口场景等比例拉伸（已修）
- 修复实施：
  1. ImguiLayer.h：加 `void DisableEditorChrome()` + `bool m_EditorChromeDisabled` flag
  2. ImguiLayer.cpp：Begin()/End() 加 `if (m_EditorChromeDisabled) return;`；加 DisableEditorChrome() 实现（设 flag + IniFilename=nullptr + ClearIniSettings 清已加载布局）

## Bug 7（疑似）：Snap 最大化后 UI 消失 — DPI 代码导致 glViewport 超出 framebuffer
- 现象：用户 4K 屏，Snap 到 (3173,458) 看得到 UI，Snap 最大化到 (3840,2054) 看不到 UI
- 疑似根因：用户加的 DPI 代码（GameLayer.cpp L105-114）每帧设 `DisplayFramebufferScale = glfwGetWindowContentScale`
  - 4K 屏即使 Windows 设 100% DPI，glfwGetWindowContentScale 可能返回 2.0
  - imgui_impl_opengl3 RenderDrawData 用 `glViewport = DisplaySize * FramebufferScale`（imgui_impl_opengl3.cpp:462）
  - 若 scale=2：glViewport = 3840*2 = 7680 > 实际 framebuffer 3840 → UI 画到 framebuffer 外，不显示
- 处理：注释掉 DPI 代码（GameLayer.cpp L105-114），加诊断日志打印 window/spec/fb/scale
- 待用户打包测 3840x2054：若 UI 恢复 → 确认 DPI 代码是根因；若仍消失 → 看 spec vs fb 日志定位
- 长期正确做法：若要支持高 DPI，应直接用 glfwGetFramebufferSize 作为 DisplaySize（而非 window size * scale）
